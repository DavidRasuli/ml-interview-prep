📋 How to Answer ML Deployment Scenarios

A systematic approach to tackling deployment scenarios in interviews. Learn the framework for breaking down problems and communicating your thinking clearly.

The 5-Step Framework
When faced with any ML deployment scenario, follow this systematic approach. Interviewers care more about your thought process than getting the "perfect" answer.
Step 1
Clarify Requirements & Constraints

Goal: Make sure you understand the problem before jumping to solutions.

What to do:

  • Restate the scenario in your own words
  • Identify key constraints (budget, latency, traffic patterns)
  • Calculate basic requirements (VRAM needed, expected QPS)
  • Ask clarifying questions if anything is ambiguous
Example:

"Let me make sure I understand: We're deploying a 7B parameter LLM for a chatbot serving ~100 concurrent users with bursty traffic. The model is trained, and we need it production-ready. Is that correct?"

"First, let me calculate the resources needed: 7B parameters in FP16 = 14GB VRAM minimum..."

Step 2
Identify the Core Trade-offs

Goal: Show you understand there's no single "right" answer - only trade-offs.

Common trade-offs in ML deployment:

  • Speed vs Cost: Faster instances cost more
  • Accuracy vs Speed: Lower precision (FP16→FP8) = faster but less accurate
  • Time-to-Market vs Optimization: Deploy fast vs deploy optimized
  • Simplicity vs Performance: FastAPI vs Triton/vLLM
  • Flexibility vs Lock-in: ONNX (portable) vs TensorRT (NVIDIA-only but faster)
Example:

"The key trade-off here is between speed-to-market and production-ready performance. We could deploy the base PyTorch model today, but it'll be slower. Or we could spend 2-3 days optimizing to TensorRT first for better performance from day one."

Step 3
Propose Your Solution with Reasoning

Goal: Make a concrete recommendation and explain WHY.

Structure your answer:

  • State your recommendation clearly
  • Explain the reasoning (show your work!)
  • Acknowledge what you're trading off
  • Quantify when possible (costs, performance, time)
Example:

"I'd recommend deploying on g5.xlarge with vLLM. Here's why:"

1. VRAM: 24GB gives us room for the 14GB model plus 10GB headroom for batching and KV cache

2. Serving: vLLM is purpose-built for LLMs, handles batching and KV cache automatically

3. Cost: One g5.xlarge can handle 100 concurrent users efficiently through batching (~$720/month)

The trade-off: We're paying 2x per instance vs g4dn, but we need fewer instances overall."

Step 4
Explain How You'd Validate Success

Goal: Show you think about operations, not just deployment.

Cover these areas:

  • Functional: Does it respond correctly?
  • Performance: Are latency and throughput acceptable?
  • Resources: Is GPU utilization healthy (60-85%)?
  • Reliability: What's the error rate?
  • Monitoring: What metrics would you track?
Example:

"I'd validate the deployment by checking:"

1. GPU memory usage: Should be 16-19 GB (model loaded correctly)

2. Latency: p95 should be <100ms for base PyTorch, <20ms after TensorRT optimization

3. Batching: Average batch size should be 8-32 (confirms batching is working)

4. Load test: 100 concurrent users for 10 minutes with <0.5% error rate

Step 5
Discuss Next Steps & Iterations

Goal: Show you think beyond the immediate solution.

What to consider:

  • What would you optimize next?
  • How would you handle growth (10x traffic)?
  • What failure modes should you prepare for?
  • How would you improve this over time?
Example:

"For next steps, I'd:"

1. Week 1: Deploy base PyTorch model, validate with real users

2. Week 2: Optimize to TensorRT in parallel, benchmark performance

3. Week 3: Swap in TensorRT version for 5-10x speedup

4. Ongoing: Monitor for model drift, track SOTA models, consider scaling to multiple regions if traffic grows

Do's and Don'ts
✅ Do
  • Think out loud - show your reasoning
  • Ask clarifying questions
  • Quantify when possible (costs, latency, VRAM)
  • Acknowledge trade-offs explicitly
  • Reference specific tools (vLLM, TensorRT, g5.xlarge)
  • Explain WHY you chose something
  • Admit what you don't know
  • Connect to your infrastructure experience
❌ Don't
  • Jump to a solution without understanding the problem
  • Say "it depends" without explaining what it depends on
  • Use vague terms ("fast", "efficient") without numbers
  • Ignore costs or business constraints
  • Pretend to know something you don't
  • Give only one option without discussing alternatives
  • Forget about operations/monitoring
  • Overcomplicate simple problems
Communication Tips

Structure Your Verbal Response

Opening: "Let me think through this systematically..."

During: "First, the requirements... Next, the trade-offs... So my recommendation would be..."

Closing: "Does that make sense? Happy to dive deeper into any part."

Use Numbers to Show Confidence

❌ "We'd need a GPU with enough memory"

✅ "We'd need at least 16GB VRAM since the model is 14GB in FP16, plus ~4GB for activations and batching"

❌ "vLLM would be more efficient"

✅ "vLLM typically achieves 2-3x better throughput than Triton for LLM workloads through continuous batching"

Connect to Your Background

Since you have strong infrastructure experience, draw parallels:

"This is similar to choosing instance types for web services - we want enough headroom for traffic spikes but don't want to overpay for unused capacity."

"I've dealt with similar scaling challenges deploying microservices on Kubernetes - the batching optimization here is like connection pooling in databases."

Example: Putting It All Together
Here's how a complete answer flows using the 5-step framework:

Interviewer: "We need to deploy a 7B LLM for a chatbot serving about 100 concurrent users. How would you approach this?"

You: "Let me think through this systematically. [Step 1: Clarify] So we have a trained 7B model for chat with about 100 concurrent users, and we need it production-ready. First, let me calculate the resources: 7B parameters in FP16 is 14GB, but we need extra VRAM for activations and batching - probably 18-20GB total."

"[Step 2: Trade-offs] The key decision is instance type. T4 (16GB VRAM) is cheaper but too tight - we'd be limited to batch_size=1. A10G (24GB VRAM) costs 2x more per instance but enables batching of 16-32 requests."

"[Step 3: Solution] I'd go with g5.xlarge (A10G). Even though it's $1/hour vs $0.50/hour for g4dn, the batching efficiency means we only need one instance instead of 2-3. Total cost is actually lower at $720/month."

"For serving, I'd use vLLM since it's purpose-built for LLMs - handles KV cache and continuous batching automatically, which is critical for chat workloads."

"[Step 4: Validation] To validate, I'd check GPU memory is 16-19 GB (model loaded correctly), latency p95 <100ms, batch size averaging 8-32, and run a load test with 100 concurrent users for 10 minutes."

"[Step 5: Next Steps] I'd deploy the base PyTorch model first to validate the use case, then optimize to TensorRT in parallel for 5-10x speedup. That way we're in production fast while still getting to optimal performance."

"Does that approach make sense? Happy to dive deeper into any part."

💡 Common Questions & Deep Dives

Detailed explanations answering the "why" behind ML deployment concepts. Goes beyond definitions to build real understanding.

Why does floating point (FP) only handle numbers? What if the value was a string?

Neural networks can ONLY work with numbers. Everything must be converted to numbers before the model can process it.

Why? Because neural networks are fundamentally just mathematical operations:

  • Matrix multiplication (numbers × numbers)
  • Addition (numbers + numbers)
  • Activation functions (mathematical operations on numbers)

What About Text (Strings)?

Text gets converted to numbers first through a process called tokenization and embedding:

"Hello world" ↓ (tokenization) [15496, 995] ← Token IDs (numbers representing words) ↓ (embedding) [0.234, -0.891, 0.456, ...] ← 768-dimensional vector of floats ↓ Model processes these floating-point numbers

Examples of Data Conversion:

Images: Converted to pixel values (0-255), normalized to floats (0.0-1.0)

Audio: Converted to waveform amplitudes (floats)

Categories: "Cat", "Dog", "Bird" → [0, 1, 2] or one-hot vectors

Bottom line: Everything becomes floating-point numbers before the model sees it. This is why FP precision (FP32, FP16, FP8) matters - it's the fundamental data type neural networks operate on.

Why does training take weeks and cost millions of dollars?

Time: Why Weeks?

Training a 7B parameter model from scratch requires processing trillions of data points:

Actual Timeline Example (Llama-2-7B):

• Trained on 2 trillion tokens (words/pieces of words)

• Used ~180,000 GPU hours total

• On 2,048 GPUs running simultaneously = ~88 hours per GPU

• That's 3.7 days of perfect utilization

• In reality: ~2 weeks accounting for checkpointing, debugging, restarts

Each token requires:

  • A forward pass (compute prediction) - touching all 7 billion parameters
  • A backward pass (calculate gradients, update weights)
  • This happens billions of times

Cost: Why Millions?

Hardware Cost Breakdown for Training a 7B Model: Hardware: NVIDIA A100 GPUs (80GB) Cloud cost: ~$2.50/hour per GPU GPUs needed: ~2,048 GPUs (for reasonable training time) Math: 2,048 GPUs × $2.50/hour × 88 hours = $450,560 Additional costs: • Data storage and processing: ~$50,000 • Engineer time (months of work): ~$200,000 • Failed experiments/iterations: 2-3x the above • Infrastructure and overhead: ~$100,000 Total: $1-3 million for a 7B model from scratch

For Larger Models (70B, GPT-4 scale):

• 10-100x more compute required

• Total costs: $10-100+ million

• GPT-4 training estimated at $100+ million

This is why:

  • Companies use pre-trained models from HuggingFace (cost already paid)
  • Fine-tuning is preferred (hours, thousands of dollars vs millions)
  • Your job focuses on deploying trained models, not training them
  • Understanding these costs helps justify infrastructure optimization decisions
TensorRT makes things 2-10x faster - but faster at what? Training or serving?

Serving (inference) ONLY. TensorRT doesn't help with training at all.

Why the Speedup for Inference

Before TensorRT (Base PyTorch) After TensorRT Optimization
• Many small GPU operations
• Each operation has overhead
• Running in FP32 (slow)
• Generic kernels
• Result: 100ms per request 		• Operations fused together
• Running in FP16 (2x faster)
• Custom GPU kernels
• Memory optimized
• Result: 10-20ms per request

Where the Speedup Comes From:

Three Main Optimizations:

1. Precision Reduction (2x speedup): FP32 → FP16 means GPUs can do twice as many operations per second

2. Layer Fusion (2-3x speedup): Combining multiple operations reduces GPU kernel launches (which have overhead)

3. Kernel Optimization (1.5-2x speedup): Generate GPU code specifically optimized for your model and GPU architecture

Combined: 2x × 2.5x × 1.5x ≈ 7.5x total speedup

Why Training Doesn't Benefit:

  • Needs high precision: Training requires FP32 for numerical stability during gradient updates
  • Already optimized: Training frameworks already batch and optimize efficiently
  • Different workload: TensorRT's optimizations are inference-specific (frozen weights, no gradients)

Key takeaway: TensorRT is your go-to optimization for production inference on NVIDIA GPUs. It's a deployment tool, not a training tool.

Why is request batching optimal for GPU efficiency?

This is one of the most important concepts for understanding ML infrastructure costs and performance!

🚌 The Bus Analogy

Without batching (1 request at a time):

Bus arrives → picks up 1 passenger → drives route → returns
Bus arrives → picks up 1 passenger → drives route → returns
Result: Bus is mostly empty. Lots of trips. Inefficient.

With batching (many requests together):

Bus waits → picks up 50 passengers → drives route → returns
Result: Bus is full. Fewer trips. Much more efficient.

The Technical Reality

GPUs are massively parallel processors with thousands of cores that can work simultaneously.

NVIDIA A10G GPU (in g5 instances):

• 9,216 CUDA cores available

• Can perform 9,216 operations at the same time

• Processing 1 request uses only 100-500 cores

• Other 8,700+ cores sit IDLE (wasted money!)

The Dramatic Impact

Metric Without Batching (1 req) With Batching (32 reqs)
GPU Utilization 5-10% 80-90%
Throughput 10 req/s 300+ req/s
Latency per request ~50ms ~60ms (minimal increase)
Cost efficiency Baseline 30x better

Real Cost Impact:

Scenario: Need to serve 300 requests per second

Without batching: Need 30 GPUs @ $1/hour = $30/hour = $21,600/month

With batching: Need 1 GPU @ $1/hour = $1/hour = $720/month

Savings: $20,880/month (96% cost reduction!)

The Magic Explained

GPUs can process a batch of 32 inputs almost as fast as 1 input because they execute all 32 in parallel using different cores.

Single request: [0.1, 0.2, 0.3, ...] → Model → [0.8, 0.1, 0.1] Uses 500 cores, takes 50ms Batch of 32 requests: [Request 1: [0.1, 0.2, ...]] ┐ [Request 2: [0.5, 0.3, ...]] │ [Request 3: [0.2, 0.8, ...]] ├→ All processed in parallel ... │ [Request 32: [0.9, 0.1, ...]] ┘ Uses 9,000 cores, takes 60ms (only 20% slower for 32x throughput!)

The Trade-off

Cost: Need to wait a bit to collect a batch (adds ~10-50ms latency)

Benefit: 10-30x better throughput and cost efficiency

Verdict: Almost always worth it for production systems

Tools like Triton and vLLM handle batching automatically - they collect incoming requests, batch them intelligently, and maximize GPU utilization without you having to code it yourself.

Is there a standard ML deployment lifecycle like CI/CD? What's the proper terminology?

Yes! The term is MLOps (Machine Learning Operations) - ML's equivalent to DevOps.

MLOps is relatively new (coined around 2015-2017) but rapidly standardizing across the industry.

The MLOps Lifecycle

1. Data Engineering ↓ Collect, clean, version data ↓ Tools: DVC, feature stores 2. Model Development ↓ Experiment, train, evaluate ↓ Tools: MLflow, Weights & Biases 3. Model Validation ↓ Test accuracy, bias, performance ↓ A/B testing framework 4. Model Deployment ← YOUR FOCUS AREA ↓ Optimize (ONNX, TensorRT) ↓ Package and deploy ↓ Tools: Docker, Kubernetes, Triton 5. Model Serving ← YOUR FOCUS AREA ↓ API endpoints, load balancing ↓ Tools: Triton, vLLM, FastAPI 6. Monitoring ← YOUR FOCUS AREA ↓ Track performance, drift, errors ↓ Tools: Prometheus, Grafana 7. Model Governance ↓ Versioning, rollback, compliance ↓ Tools: MLflow Model Registry 8. Continuous Training (CT) ↓ Retrain on new data ↓ Like CI/CD but for models

MLOps vs Traditional CI/CD

Aspect Software CI/CD ML MLOps
Primary artifact Code Code + Data + Model
Testing Unit tests, integration tests All that + accuracy tests, bias tests, A/B tests
Deployment Deploy once, runs indefinitely Model degrades over time (drift)
Versioning Git for code Git + data versioning + model registry
Continuous process CI/CD CI/CD/CT (Continuous Training)

Key Differences from Traditional Software

1. Models Drift Over Time

Software doesn't suddenly break just because time passed. Models degrade as the world changes and data distributions shift. You need continuous monitoring and periodic retraining.

2. Experiments Are First-Class Citizens

In ML, you might train 100 different models before finding the right one. You need to track: which model version, which data, which hyperparameters, which performance. Way more complex than git commits.

3. Data Versioning Is As Critical As Code Versioning

You can't reproduce a model without the exact training data. "The model worked yesterday" isn't enough - you need to know which data version was used.

4. A/B Testing Is Standard

You can't just "release" a new model like you release code. You need to compare it against the old model with real traffic to see if it's actually better in production.

Your Role in MLOps

When working through the scenarios, you're focusing on:

  • Model Deployment: Optimize, package, deploy (ONNX, TensorRT, Docker)
  • Model Serving: Infrastructure, scaling, instance selection
  • Monitoring: Latency, errors, drift detection

This is one critical piece of the larger MLOps lifecycle, and it's a full-time specialized role: ML Engineer / MLOps Engineer / ML Infrastructure Engineer.

In interviews: Understanding where your work fits in the bigger MLOps picture shows maturity and helps you ask good questions about the team's workflow.

How do I validate that my model deployment is working correctly in production?

Deploying a model is only half the job - you need to validate it's actually working correctly. Here's a systematic approach across multiple dimensions:

1. Functional Validation (Does it work at all?)

Basic API Test:

Send a simple test request to your deployed model:

curl http://your-instance:8000/v1/chat/completions -d '{"messages": [{"role": "user", "content": "What is 2+2?"}]}'

✅ Expected: Coherent response like "2+2 equals 4"

❌ Bad: Gibberish, timeout, or error response

2. Performance Validation (Is it fast enough?)

Latency measurement: Time 100 requests and calculate percentiles (p50, p95, p99)

Expected Latency:

PyTorch FP16: ~50-100ms (p95)

TensorRT FP16: ~10-20ms (p95)

Warning sign: >200ms consistently means something's misconfigured

Throughput measurement: Send 100 concurrent requests, measure requests/second

3. Resource Utilization (Is hardware used efficiently?)

# Check GPU metrics nvidia-smi ✅ Healthy Metrics: GPU Memory: 16-19 GB / 24 GB (66-79%) GPU Utilization: 60-85% under load Temperature: 60-80°C Power Usage: 150-250W (A10G max: 300W) ❌ Warning Signs: GPU Memory: >23 GB (risk of OOM crash) GPU Utilization: <20% (not batching, wasting money) Temperature: >85°C (cooling issue)

4. Batching Validation (Is batching working?)

Check your serving platform's metrics (vLLM or Triton):

Good Batching Indicators:

Average batch size: 8-32 (requests being combined efficiently)

GPU utilization: Spikes to 80-90% when processing batches

Queue length: 0-10 requests waiting

Bad: Batch size of 1-2 means batching isn't working - you're wasting 90% of your GPU!

5. Error Rate Monitoring (Is it reliable?)

Metric Healthy Unhealthy
Success rate (200 OK) >99.5% <95%
Server errors (5xx) <0.2% >1%
Timeouts <0.1% >2%
Request queue 0-10 >50

6. Load Testing (Can it handle realistic traffic?)

Sustained load test: Simulate 100 concurrent users for 10 minutes

What to Watch:

✅ Latency stays stable over time (no memory leaks)

✅ Error rate stays below 0.5%

✅ No OOM crashes

✅ GPU memory stays below 23 GB

Burst test: Simulate traffic spike (500 requests in 30 seconds) to test bursty traffic handling

The 30-Minute Validation Checklist

If you only have limited time, check these essentials:

Tier 1 - Must Check (10 min):

1. Send test request → get coherent response

2. GPU memory: 16-19 GB used (model loaded properly)

3. Latency: p95 meets requirements

4. No 5xx errors on test requests

Tier 2 - Should Check (10 min):

5. GPU utilization: 60-85% under load

6. Average batch size: 8-32

7. Load test: 100 concurrent requests for 5 min, no crashes

Tier 3 - Nice to Check (10 min):

8. GPU temperature under 85°C

9. Queue length stays under 20

10. Integration with your actual application works

Key insight: A "deployed" model that crashes under load, has 5% error rate, or uses only 10% of GPU isn't really production-ready. Validation ensures you're not just running, but running correctly and efficiently.

Why This Matters for ML

Neural networks are just massive matrix multiplications. A single matrix operation might involve millions of multiplications that can all happen simultaneously. CPUs do these sequentially (even with 8-16 cores). GPUs do thousands at once, making them 10-100x faster for ML workloads. This is why a $1,000 GPU outperforms a $10,000 CPU for inference.

4. Precision Formats: Speed vs Accuracy Trade-off
Lower precision means less memory, faster computation, but slightly reduced accuracy. FP16 is the sweet spot for most production deployments - 2x speed boost with negligible accuracy loss.
Memory Size Comparison for 7B Parameter Model
FP32
32 bits per parameter (4 bytes)
Model Size: 28 GB
Speed: Baseline (1x) | Accuracy: Highest
FP16
16 bits per parameter (2 bytes) ⭐ Production Standard
Model Size: 14 GB
Speed: 2x faster | Accuracy: ~99% of FP32
BF16
16 bits per parameter (2 bytes) - Google's format
Model Size: 14 GB
Speed: 2x faster | Accuracy: Better range than FP16
FP8
8 bits per parameter (1 byte) - Cutting Edge
Model Size: 7 GB
Speed: 4x faster | Accuracy: ~95% of FP32

🎯 Decision Framework for Production

FP32: Almost never used for inference. Too slow, too large. Only for training when stability is critical.

FP16: ⭐ The sweet spot. Use this by default. 2x speed, half the VRAM, negligible accuracy loss. Standard for production.

BF16: Same benefits as FP16 but better numerical range. Good alternative to FP16, especially for newer models designed for it.

FP8: Cutting edge. Only use if you have hardware support (H100, L4 GPUs) and models designed for it. 4x speedup but noticeable accuracy trade-off.

TensorRT handles conversion automatically - you specify target precision, it optimizes the model accordingly.

7. Inference vs Training: Two Different Workflows
Training and inference are fundamentally different operations with different goals, costs, and optimization strategies. Your role focuses entirely on inference.
🎓 Training
Goal:
Learn patterns from data by adjusting billions of parameters
Process:
1. Forward pass (predict)
2. Calculate loss (error)
3. Backward pass (gradients)
4. Update weights
Repeat billions of times
Time & Cost:
⏱️ Weeks to months
💰 $1-100+ million
For 7B model: ~2 weeks on 2,048 GPUs
Precision:
FP32 required for stability
Needs high precision for gradient updates
Who does it:
Data scientists, ML researchers
NOT your role
🚀 Inference
Goal:
Use trained model to make predictions on new data
Process:
1. Load frozen model
2. Forward pass only
3. Return prediction
No weight updates
Time & Cost:
⏱️ Milliseconds per request
💰 Pennies per 1000 requests
For 7B model: ~20-50ms per request
Precision:
FP16/BF16/FP8 for speed
Can use lower precision safely
Who does it:
ML Engineers, MLOps teams
THIS IS YOUR ROLE!

🎯 Key Differences That Matter for Your Work

1. Frequency: Training happens once (or occasionally). Inference happens millions of times per day. This is why optimizing inference is critical.

2. Optimization focus: Training optimizes for accuracy. Inference optimizes for speed and cost.

3. Hardware: Training can use FP32 and take time. Inference must be fast, so we use FP16/FP8, batching, TensorRT, etc.

4. Your role: You receive trained models (.pt files) from data scientists and optimize them for production inference. You never train models yourself.

Interview tip: When discussing optimizations (batching, precision, TensorRT), always clarify you're talking about inference, not training!

🎯 The 30-Minute Validation Checklist

Tier 1 - Must Check (10 minutes):

1. Send test request → get coherent response

2. GPU memory: 16-19 GB used (model loaded properly)

3. Latency: p95 meets requirements

4. No 5xx errors on test requests

Tier 2 - Should Check (10 minutes):

5. GPU utilization: 60-85% under load

6. Average batch size: 8-32

7. Load test: 100 concurrent requests for 5 min, no crashes

Tier 3 - Nice to Check (10 minutes):

8. GPU temperature under 85°C

9. Queue length stays under 20

10. Integration with your application works

⚠️ Common Failure Modes to Watch For

OOM (Out of Memory) Crashes: GPU memory >23 GB → service crashes randomly under load

Not Batching: Batch size ~1, GPU utilization <30% → wasting 90% of GPU, need 10x more instances

Memory Leaks: GPU memory creeps up over hours → eventually crashes, needs restart

Queue Buildup: Queue length keeps growing → can't keep up with traffic, need more instances

Thermal Throttling: GPU >85°C → performance degrades automatically to cool down

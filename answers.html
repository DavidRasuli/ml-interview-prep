✅ Scenario 1: Greenfield Deployment - Complete Answer

Step-by-step solution with your learning points highlighted in red

The Question

You're deploying a 7B parameter LLM for a new internal chatbot. The model has been trained and handed off by your data science team. You expect about 100 concurrent users with bursty traffic (chat usage during work hours). There's no existing ML infrastructure. Design the deployment from scratch.

Step 1: Calculate VRAM & Choose Instance

Calculation:

7B parameters × 2 bytes (FP16) = 14 GB model
+ Activations (~2 GB)
+ Batching buffer (~1 GB)
+ KV cache (~2-4 GB)
= 18-20 GB total needed

✅ Decision: g5.xlarge (A10G, 24GB VRAM)

• 10GB headroom enables batch sizes 16-32
• $720/month but needs only ONE instance due to batching
• g4dn (16GB) would need 2-3 instances = $900-1350/month

❌ Lessons Learned

Never calculate just the model size! You learned that 14GB model ≠ 14GB VRAM needed. Always add ~30-40% overhead for activations, batching, and KV cache.

Don't optimize for per-hour cost. g4dn is cheaper per hour but worse total cost. Always calculate: (instances needed) × (cost per instance).

Step 2: Optimization Strategy
✅ Hybrid Approach

Week 1: Deploy PyTorch FP16 with vLLM
• Fast to market, validate use case
• 50-100ms latency acceptable for testing

Week 2-3: Optimize to TensorRT in parallel
• 5-10x speedup (10-20ms latency)
• Swap in once validated

❌ Lessons Learned

TensorRT "FP16 optimized" ≠ just using FP16. You learned that TensorRT optimization = precision conversion + Tensor Core usage + layer fusion + kernel optimization = 5-10x speedup. Simply loading PyTorch in FP16 = only 2x.

Don't block deployment on optimization. Deploy base model fast, optimize in parallel. Speed to market > perfect performance on day 1.

Step 3: Serving Platform
✅ Decision: vLLM

• Purpose-built for LLMs (your exact use case!)
• Automatic KV cache management (critical for chat)
• Continuous batching (perfect for bursty traffic)
• 2-3x better LLM throughput than Triton
• Simple one-command deployment

❌ Your Biggest Learning Point

You said: "vLLM would be used if one wants to optimize KV cache management and batching, this is not our case."

This was completely backwards!

The truth:
• Every LLM chatbot needs KV cache - it's how models remember conversation context
• 100 concurrent users absolutely requires batching - without it you need 10x more GPUs
• vLLM doesn't just "optimize" these - it handles them automatically

Correct thinking: vLLM is the DEFAULT for LLM serving, not an optional optimization. Use vLLM for LLMs, use Triton for multi-model deployments.

Step 4: Validation

Tier 1 - Must Check:

  • Send test request → coherent response
  • GPU memory: 16-19 GB / 24 GB (72-79%)
  • Latency p95 <100ms
  • No 5xx errors

Tier 2 - Should Check:

  • GPU utilization: 60-85% under load
  • Batch size averaging 8-32
  • Load test: 100 concurrent for 10 min, <0.5% errors
Complete Interview Answer

Here's how you'd deliver this in the interview:

"Let me think through this systematically. We have a 7B LLM for chat with 100 concurrent users and bursty traffic. First, VRAM calculation: 7B × 2 bytes (FP16) = 14GB, but we need ~18-20GB total for activations and batching. That means g5.xlarge with 24GB VRAM, not g4dn - even though g5 costs 2x per hour, the batching efficiency means we only need one instance vs 2-3 g4dn instances, so total cost is actually lower at $720/month."

"For serving, I'd use vLLM because it's purpose-built for LLMs - handles KV cache automatically which is critical for chat, and continuous batching handles bursty traffic perfectly. vLLM typically gets 2-3x better throughput than Triton for LLM workloads."

"My deployment strategy: Week 1, deploy the base PyTorch FP16 model to get to production fast. Week 2-3, optimize to TensorRT in parallel for 5-10x speedup, then swap it in. That way we validate the use case quickly while still getting to optimal performance."

"To validate, I'd check GPU memory is 16-19GB (model loaded correctly), latency p95 <100ms initially, batch size averaging 8-32, and run a load test with 100 concurrent users for 10 minutes with <0.5% error rate."

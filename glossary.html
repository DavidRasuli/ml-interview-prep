📖 ML Deployment Glossary

Comprehensive reference for all machine learning deployment terms and technologies

A B C F G H I L M O P R S T V
A
7B / 13B / 70B (Parameter Counts)
Model Terminology
Shorthand notation for the number of parameters in a neural network. 7B = 7 billion parameters, 13B = 13 billion, 70B = 70 billion. Each parameter is a learned weight (a single number) stored in memory.
Why it matters: Parameter count directly determines model size in memory, inference speed, and capability. A 7B model in FP16 needs ~14GB VRAM (7 billion × 2 bytes). More parameters generally means smarter but slower and more expensive. Common sizes: Small (1B-7B), Medium (13B-30B), Large (70B+).
Adapters (LoRA)
Fine-tuning Technique
A method for fine-tuning large models by adding small trainable "adapter" layers while keeping the base model frozen. LoRA (Low-Rank Adaptation) is the most popular adapter technique.
Why it matters: Instead of retraining all billions of parameters (expensive, slow), you only train tiny adapter modules. You can swap adapters to change model behavior without loading entirely new models. Much cheaper and faster than full fine-tuning.
AMX (Advanced Matrix Extensions)
Hardware Acceleration
Intel's newest CPU instruction set specifically designed for accelerating matrix multiplication operations, which are fundamental to neural network computations.
Why it matters: Makes CPU-based inference much faster for ML workloads. OpenVINO leverages AMX to optimize models for Intel CPUs. Part of the progression: SIMD → AVX-512 → AMX (each generation better for ML).
Async/Await
Python Concurrency
Python's cooperative multitasking paradigm that allows a single process to handle multiple I/O-bound operations concurrently without blocking.
Why it matters: Great for orchestration tasks (waiting for GPU, database, API calls) but doesn't solve the GIL problem for CPU-bound work. In ML serving, you'd use async for handling requests while the actual inference runs in C++ (which releases the GIL).
AVX-512 (Advanced Vector Extensions)
Hardware Acceleration
Intel's CPU instruction set that can process 512 bits (16 floats) simultaneously in a single instruction. Part of SIMD (Single Instruction, Multiple Data) family.
Why it matters: Speeds up CPU inference by processing multiple numbers at once. OpenVINO automatically uses AVX-512 when optimizing models for Intel CPUs. Makes CPU inference viable for smaller models or when GPUs are too expensive.
B
Batching (Request Batching)
Optimization Technique
Processing multiple requests simultaneously rather than one at a time. Critical for GPU efficiency because GPUs have thousands of cores that can process many inputs in parallel.
Why it matters: A GPU processing 1 request uses only 5-10% of its cores (wasteful). Processing 32 requests batched together uses 80-90% of cores and achieves 10-30x better throughput with similar latency. Essential for cost-effective serving - might need 1 GPU with batching vs 30 GPUs without. Triton and vLLM handle batching automatically.
BF16 (Brain Float 16)
Numeric Precision
Google's 16-bit floating-point format designed specifically for machine learning. Has the same range as FP32 but with reduced precision, making it less prone to overflow than standard FP16.
Why it matters: Same speed/memory benefits as FP16 (2x faster than FP32) but better numerical stability. Becoming increasingly popular, especially for training. Good middle ground between FP32 accuracy and FP16 speed.
C
C++
Programming Language
High-performance compiled language with direct hardware access. Core language for ML inference runtimes, GPU programming (CUDA), and performance-critical systems.
Why it matters: TensorRT, ONNX Runtime, and most ML serving backends are written in C++ for maximum performance. You won't write much C++ day-to-day, but you'll deploy and configure C++-based systems. Understanding this helps explain why Python + C++ bindings work so well for ML.
F
Fine-tuning
Model Training
Taking a pre-trained model and continuing to train it on your specific dataset to specialize it for your particular use case.
Why it matters: Much cheaper than training from scratch. You start with a model that already understands language/images/etc., then teach it your domain-specific knowledge. Your role would be deploying these fine-tuned models, not creating them.
FP8 (8-bit Floating Point)
Numeric Precision
Cutting-edge 8-bit floating-point format that offers 4x speed improvement and 4x memory reduction compared to FP32. Still experimental but supported by newest GPUs.
Why it matters: Bleeding edge of model optimization. NVIDIA H100 and newer GPUs have hardware support for FP8. More accuracy loss than FP16, but newer models are being designed to handle it. Can dramatically reduce serving costs.
FP16 (16-bit Floating Point)
Numeric Precision
Half-precision floating-point format that uses 16 bits instead of 32. Industry standard for production inference.
Why it matters: Sweet spot for most production deployments - 2x faster, uses half the VRAM, with negligible accuracy loss. Converting FP32→FP16 is often the first optimization step. Most GPUs have dedicated FP16 hardware (Tensor Cores on NVIDIA).
FP32 (32-bit Floating Point)
Numeric Precision
Standard single-precision floating-point format. Default for most training and initial model development. Most accurate but slowest and most memory-intensive.
Why it matters: What models start as. Your job is often converting FP32 to FP16/BF16/FP8 for production to gain speed/memory without significant accuracy loss. Understanding this tradeoff is crucial for deployment decisions.
G
g4dn (AWS Instance Type)
Cloud Infrastructure
AWS EC2 instance type with NVIDIA T4 GPUs. Older generation, cost-effective option for ML inference workloads.
Why it matters: Good for smaller models or when cost is primary concern. T4 GPUs are less powerful than newer generations but much cheaper. Common choice for development/staging or lightweight production workloads.
g5 (AWS Instance Type)
Cloud Infrastructure
AWS EC2 instance type with NVIDIA A10G GPUs. Newer generation with more VRAM and better performance than g4dn.
Why it matters: Common choice for production ML inference. Better performance and more VRAM (24GB) than g4dn, allowing larger models. Price/performance sweet spot for many workloads.
g6 (AWS Instance Type)
Cloud Infrastructure
AWS EC2 instance type with NVIDIA L4 GPUs. Newest generation optimized for inference with support for newer precision formats (FP8).
Why it matters: Most efficient for inference, especially with FP8-optimized models. Better performance per dollar for inference workloads. Hardware support for latest optimization techniques.
GIL (Global Interpreter Lock)
Python Limitation
Python's mechanism that allows only one thread to execute Python bytecode at a time, preventing true parallel execution of Python code on multiple CPU cores.
Why it matters: The biggest Python scaling challenge. Why you can't just use threading to handle millions of requests. Solutions: multiprocessing (multiple Python processes), async/await (for I/O), or rewriting in Go/Rust. Fortunately, actual ML inference (TensorRT, ONNX) runs in C++ and releases the GIL.
Go (Golang)
Programming Language
Google's compiled language designed for high-performance concurrent systems. No GIL equivalent, making it excellent for handling many simultaneous requests.
Why it matters: Often used for ML API servers, orchestration layers, and data pipelines where Python's GIL becomes a bottleneck. Faster than Python, easier than C++. Goroutines make concurrent programming straightforward. Limited ML libraries, so used alongside Python rather than replacing it.
Go Routines
Concurrency Pattern
Go's lightweight concurrency primitive. Like threads but much cheaper - you can run thousands simultaneously. No GIL equivalent means true parallelism.
Why it matters: Why Go is great for high-concurrency API servers. Each request can run in its own goroutine without the overhead of Python's GIL or process-based concurrency. Relevant when building ML serving infrastructure that needs to handle massive request volumes.
GPU (Graphics Processing Unit)
Hardware
Specialized processor originally designed for graphics but highly effective for neural network computations due to ability to perform massive parallel matrix operations. NVIDIA dominates the ML GPU market.
Why it matters: Foundation of modern ML. Can perform thousands of calculations simultaneously, making matrix operations (core of neural networks) much faster than CPUs. Has its own memory (VRAM). Most production ML models run on GPUs. Understanding GPU constraints (VRAM limits, batch sizing) is crucial for deployment.
GPU Streams (CUDA Streams)
Hardware Concept
Independent execution queues on a GPU that allow multiple operations to run concurrently. Like having multiple checkout lanes instead of one - different tasks (data transfer, inference, output copying) can overlap instead of waiting in sequence.
Why it matters: Dramatically improves GPU utilization and reduces latency. Without streams, operations are sequential (copy input → run model → copy output → repeat). With streams, multiple requests can overlap their stages, achieving 2-3x better throughput. Example: Request A runs inference while Request B copies data - both happen simultaneously. Triton and vLLM use streams automatically to maximize efficiency. Different from batching: batching combines requests in one operation, streams run multiple operations concurrently. Best practice: use both together.
H
HuggingFace
ML Platform
The GitHub of machine learning - a platform for sharing, discovering, and using pre-trained models. Also provides the transformers library for easily loading and using models.
Why it matters: Where most teams get their models. Instead of training from scratch, you pull a model from HuggingFace, fine-tune if needed, then deploy. Your team likely uses HuggingFace as the source for base models before optimization and deployment.
I
Inference
Core Concept
Using a trained model to make predictions on new data. The "running" phase of ML, as opposed to training. Your role focuses on inference - deploying and serving models efficiently to handle millions of predictions.
Why it matters: Two phases in ML: Training (teaching the model - expensive, weeks, millions of dollars, done once) and Inference (using the trained model - cheap, milliseconds, done millions of times). Users expect fast inference (<100ms for chatbots). Faster inference = serve more users on same hardware = lower costs. All optimization work (TensorRT, precision reduction, batching) targets inference speed.
L
LLM (Large Language Model)
Model Type
Neural networks with billions of parameters trained on massive text datasets. Examples: GPT-4, Claude, Llama-70B. Powerful but expensive to run.
Why it matters: These are the "expensive" models requiring significant GPU resources. Your job involves deploying these efficiently - choosing instance types, optimizing precision, managing VRAM constraints. Understanding their resource requirements is crucial for infrastructure planning.
M
Multiprocessing
Python Concurrency
Running multiple separate Python processes, each with its own GIL. Allows true parallelism but with memory overhead (each process loads the model independently).
Why it matters: One solution to Python's GIL problem. Common pattern: run N Python processes, each loads the model, distribute requests across them. Trade-off: uses N times the RAM. Often combined with async/await within each process.
MLOps (Machine Learning Operations)
Methodology
The practice of deploying, monitoring, and maintaining machine learning models in production. ML's equivalent to DevOps, encompassing the full lifecycle from data preparation through deployment and continuous monitoring.
Why it matters: Your role is MLOps-focused. The lifecycle includes: Data Engineering → Model Development → Validation → Deployment (optimize, package) → Serving (infrastructure, scaling) → Monitoring (drift, performance) → Continuous Training. Unlike traditional software, models drift over time and need retraining. Key difference from CI/CD: data versioning is as critical as code versioning.
O
ONNX (Open Neural Network Exchange)
Model Format
Vendor-neutral intermediate format for neural networks. Allows converting models from PyTorch/TensorFlow to a standard format that can run on different hardware and frameworks.
Why it matters: The "assembly language" of ML models. Convert PyTorch → ONNX → TensorRT/OpenVINO. Decouples training frameworks from inference runtimes. Widely supported. Often the first step in production optimization pipeline.
OpenVINO
Optimization Framework
Intel's toolkit for optimizing neural networks to run efficiently on Intel hardware (CPUs, integrated GPUs). Intel's equivalent of NVIDIA's TensorRT.
Why it matters: When deploying on Intel CPUs instead of NVIDIA GPUs (cost reasons, edge deployment). Automatically leverages AVX-512, AMX, and other Intel-specific acceleration. Takes ONNX models and optimizes for Intel hardware. Important for CPU-based inference scenarios.
P
Parameters (Model Parameters)
Core Concept
The learned weights in a neural network - individual numbers that the model learned during training. A 7B model has 7 billion parameters. Each parameter is stored in memory as a floating-point number.
Why it matters: Parameter count determines: (1) Model size in memory - 7B params in FP16 = 14GB VRAM needed, (2) Inference speed - more params = more computation = slower, (3) Model capability - generally more params = smarter model. This is why you see models named by their size: Llama-3-8B, GPT-3.5-175B. Critical for estimating hardware requirements and costs.
Process (vs Thread)
Concurrency Concept
Independent program execution with its own memory space. Unlike threads (which share memory and are limited by GIL in Python), processes run completely separately.
Why it matters: Python processes avoid the GIL but use more memory. In ML serving: might run 4 processes, each with its own model copy, to handle requests in parallel. Trade memory for parallelism.
PyTorch
ML Framework
The most popular Python library for building and training neural networks. Facebook/Meta's framework that's become the de facto standard for research and development.
Why it matters: Where models are born. Data scientists train in PyTorch, hand you .pt/.pth files. Your job: convert these development-optimized models into production-ready formats (ONNX, TensorRT). Understanding PyTorch helps you communicate with data science teams about model requirements and constraints.
Python
Programming Language
High-level interpreted language. Standard for ML development and data science due to rich ecosystem (PyTorch, TensorFlow, NumPy). Has performance limitations (GIL) for production serving.
Why it matters: Your primary language for ML infrastructure. Great for development, orchestration, and wrapping optimized C++ inference engines. The GIL creates scaling challenges, so you'll use multiprocessing, async, or other languages (Go/Rust) for high-throughput serving layers.
R
RAM (Random Access Memory)
Hardware
System memory accessible by the CPU. General-purpose storage for running programs and data.
Why it matters: Different from VRAM (GPU memory). When running models on CPU or loading models before transferring to GPU, you need sufficient RAM. Important for understanding instance sizing and multiprocessing constraints (each process needs RAM for model copy).
Rust
Programming Language
Modern systems programming language focused on safety and performance. Memory-safe without garbage collection, as fast as C++, no GIL equivalent.
Why it matters: Increasingly used for performance-critical ML infrastructure. Parts of vLLM are written in Rust. Safer than C++, faster than Go. Harder to write than both. Used for components where safety and performance are both critical (serving layers, data pipelines).
S
SIMD (Single Instruction, Multiple Data)
Hardware Concept
CPU capability to perform the same operation on multiple data points simultaneously. General category that includes AVX-512, AMX, and other vectorized instruction sets.
Why it matters: Foundation of CPU-based ML acceleration. Instead of processing numbers one at a time, process many at once. Understanding this helps explain why CPU inference isn't always terrible - modern CPUs can be surprisingly effective for smaller models when using SIMD properly.
SLM (Small Language Model)
Model Type
Language models with fewer parameters (hundreds of millions to low billions). Examples: Llama-3B, Phi-3. Much faster and cheaper to run than LLMs.
Why it matters: Not everything needs GPT-4. For specific tasks (classification, simple generation), SLMs are good enough and much cheaper. Can run on CPUs or smaller GPUs. Part of your job is choosing the right model size for the task - don't overspend on capability you don't need.
SOTA (State of the Art)
ML Concept
The best-performing model currently available for a specific task. Changes frequently as new models are released.
Why it matters: Part of your role is tracking when better models become available. Benchmarks change constantly. You might need to re-deploy when SOTA shifts. Understanding evaluation metrics helps you decide when upgrading models is worth the migration effort.
T
TensorRT
Optimization Framework
NVIDIA's optimization toolkit that takes trained models (PyTorch, ONNX) and optimizes them specifically for NVIDIA GPUs. Performs layer fusion, precision calibration, and kernel auto-tuning.
Why it matters: Makes models 2-10x faster on NVIDIA GPUs. Standard optimization step for production deployment on NVIDIA hardware. Works by analyzing your model and generating highly optimized GPU kernels. Often combined with precision reduction (FP32→FP16).
Threading
Python Concurrency
Running multiple threads within a single Python process. Due to GIL, threads don't provide true parallelism for CPU-bound work in Python.
Why it matters: Often misunderstood - threading in Python is good for I/O-bound work (network, disk) but doesn't help with CPU-bound computation due to GIL. For ML serving, the inference itself (C++ code) releases the GIL, so threading can work, but multiprocessing or async/await are more common patterns.
TPU (Tensor Processing Unit)
Hardware
Google's custom-designed chips specifically optimized for tensor operations (neural network computations). Only available on Google Cloud Platform.
Why it matters: Alternative to NVIDIA GPUs. Sometimes faster, sometimes cheaper, but locked to Google Cloud. If your company uses GCP, TPUs might be an option. Understanding TPU vs GPU tradeoffs helps with cloud strategy decisions.
Training (Model Training)
Core Concept
The process of teaching a neural network by showing it millions/billions of examples and adjusting parameters to minimize errors. Extremely expensive and time-consuming compared to inference.
Why it matters: Training a 7B model from scratch takes ~2 weeks on 2,048 GPUs and costs $1-3 million. Larger models (GPT-4 scale) cost $10-100+ million. This is why: (1) Companies use pre-trained models from HuggingFace instead of training from scratch, (2) Fine-tuning (cheaper, hours, thousands) is preferred over full training, (3) Your role focuses on inference/deployment, not training. Understanding training costs helps justify infrastructure decisions.
Triton Inference Server
Serving Platform
NVIDIA's production-grade inference server. Supports multiple model formats (PyTorch, ONNX, TensorRT), handles request batching, provides HTTP/gRPC APIs, and includes monitoring/metrics.
Why it matters: Industry-standard serving solution for NVIDIA deployments. Like nginx but for ML models. Handles the hard parts: request batching (crucial for GPU efficiency), concurrent requests, model versioning, health checks. You configure and deploy Triton rather than building your own serving layer.
V
vLLM
Serving Platform
Specialized high-performance serving system specifically for Large Language Models. Uses PagedAttention and other LLM-specific optimizations to maximize throughput.
Why it matters: When serving LLMs specifically, vLLM is often faster than general-purpose Triton. Optimized for text generation workloads. Handles the unique challenges of LLM serving (KV cache management, long sequences, batching strategies). Choose vLLM for LLMs, Triton for other model types.
VRAM (Video RAM)
Hardware
GPU memory. Much faster than system RAM for GPU operations but limited in size and expensive. Models must fit in VRAM to run on GPU.
Why it matters: The primary constraint in GPU deployments. A 70B parameter model in FP16 needs ~140GB VRAM. This drives instance selection (g5 has 24GB, might need multiple GPUs or model quantization). Understanding VRAM requirements is crucial for infrastructure planning and cost optimization.

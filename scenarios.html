📌 Assumptions for All Scenarios

Scenario 1: Greenfield Deployment
Beginner

You're deploying a 7B parameter LLM for a new internal chatbot. The model has been trained and handed off by your data science team. You expect about 100 concurrent users with bursty traffic (chat usage during work hours). There's no existing ML infrastructure.

Your task: Design the deployment from scratch. What instance type, serving platform, and optimization strategy would you choose?

Step 1: Calculate VRAM & Choose Instance

Calculation:

7B parameters × 2 bytes (FP16) = 14 GB model
+ Activations (~2 GB)
+ Batching buffer (~1 GB)
+ KV cache (~2-4 GB)
= 18-20 GB total needed

✅ Decision: g5.xlarge (A10G, 24GB VRAM)

Reasoning:

  • 10GB headroom enables batch sizes 16-32
  • $720/month but needs only ONE instance due to batching
  • g4dn (16GB) would need 2-3 instances = $900-1350/month

Rule: Never deploy with <20% VRAM headroom

❌ Common Mistakes

Mistake #1: Thinking 14GB model needs exactly 14-16GB VRAM

This ignores activations, batching, KV cache. You'd get OOM crashes!

Mistake #2: Choosing g4dn because it's "cheaper per hour"

Without batching, you need 3x instances. Total cost is HIGHER.

✅ Correct: Calculate total VRAM (model + overhead), leave 20%+ headroom, optimize for total cost.

Step 2: Optimization Strategy
✅ Hybrid Approach

Week 1: Deploy PyTorch FP16 with vLLM

  • Fast to market, validate use case
  • 50-100ms latency acceptable for testing

Week 2-3: Optimize to TensorRT in parallel

  • 5-10x speedup (10-20ms latency)
  • Swap in once validated
Step 3: Serving Platform
✅ Decision: vLLM

Why vLLM is perfect:

  • Purpose-built for LLMs
  • Automatic KV cache management (critical for chat!)
  • Continuous batching (handles bursty traffic)
  • 2-3x better LLM throughput than Triton
❌ Your Original Mistake (Biggest Learning)

You said: "vLLM would be used if one wants to optimize KV cache management and batching, this is not our case."

This was completely backwards!

The truth:

  • Every LLM chatbot needs KV cache (conversation context!)
  • 100 concurrent users requires batching (or need 10x GPUs)
  • vLLM handles these automatically

✅ Correct understanding: vLLM is the DEFAULT for LLM serving, not an optional optimization!

Step 4: Validation

Tier 1 - Must Check:

  • Send test request → coherent response
  • GPU memory: 16-19 GB / 24 GB (72-79%)
  • Latency p95 <100ms
  • No 5xx errors

Tier 2 - Should Check:

  • GPU utilization: 60-85% under load
  • Batch size averaging 8-32
  • Load test: 100 concurrent for 10 min

Advanced

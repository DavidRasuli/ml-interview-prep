Scenario 1: Greenfield Deployment
Beginner

You're deploying a 7B parameter LLM for a new internal chatbot. The model has been trained and handed off by your data science team. You expect about 100 concurrent users with bursty traffic (chat usage during work hours). There's no existing ML infrastructure.

Your task: Design the deployment from scratch. What instance type, serving platform, and optimization strategy would you choose?

Step 1: Calculate VRAM & Choose Instance

Calculation:

7B parameters × 2 bytes (FP16) = 14 GB model
+ Activations (~2 GB)
+ Batching buffer (~1 GB)
+ KV cache (~2-4 GB)
= 18-20 GB total needed

✅ Decision: g5.xlarge (A10G, 24GB VRAM)

Reasoning:

  • 10GB headroom enables batch sizes 16-32
  • $720/month but needs only ONE instance due to batching
  • g4dn (16GB) would need 2-3 instances = $900-1350/month

Rule: Never deploy with <20% VRAM headroom

❌ Common Mistakes

Mistake #1: Thinking 14GB model needs exactly 14-16GB VRAM

This ignores activations, batching, KV cache. You'd get OOM crashes!

Mistake #2: Choosing g4dn because it's "cheaper per hour"

Without batching, you need 3x instances. Total cost is HIGHER.

✅ Correct: Calculate total VRAM (model + overhead), leave 20%+ headroom, optimize for total cost.

Step 2: Optimization Strategy
✅ Hybrid Approach

Week 1: Deploy PyTorch FP16 with vLLM

  • Fast to market, validate use case
  • 50-100ms latency acceptable for testing

Week 2-3: Optimize to TensorRT in parallel

  • 5-10x speedup (10-20ms latency)
  • Swap in once validated
Step 3: Serving Platform
✅ Decision: vLLM

Why vLLM is perfect:

  • Purpose-built for LLMs
  • Automatic KV cache management (critical for chat!)
  • Continuous batching (handles bursty traffic)
  • 2-3x better LLM throughput than Triton
❌ Your Original Mistake (Biggest Learning)

You said: "vLLM would be used if one wants to optimize KV cache management and batching, this is not our case."

This was completely backwards!

The truth:

  • Every LLM chatbot needs KV cache (conversation context!)
  • 100 concurrent users requires batching (or need 10x GPUs)
  • vLLM handles these automatically

✅ Correct understanding: vLLM is the DEFAULT for LLM serving, not an optional optimization!

Step 4: Validation

Tier 1 - Must Check:

  • Send test request → coherent response
  • GPU memory: 16-19 GB / 24 GB (72-79%)
  • Latency p95 <100ms
  • No 5xx errors

Tier 2 - Should Check:

  • GPU utilization: 60-85% under load
  • Batch size averaging 8-32
  • Load test: 100 concurrent for 10 min
Scenario 2: Model Upgrade Same Hardware
Beginner

Your 7B chatbot has been running successfully on g5.xlarge for 3 months. The data science team trained a better model (same 7B size, better quality). You need to deploy the new model with zero downtime and be able to roll back if issues arise.

Your task: Plan the upgrade strategy. How do you deploy the new model, validate it, and ensure you can rollback safely?

Step 1: Clarify Requirements & Constraints

What we know:

  • Same model size (7B) → Same VRAM requirements (18-20GB total)
  • Currently in production serving real users
  • Zero downtime required
  • Must be able to rollback quickly

Assumptions:

  • Same 100 concurrent users from Scenario 1
  • Business hours traffic (9am-5pm peak)
  • Rollback within minutes if critical issues found
✅ Key Insight: Can't Run Both Models on Same GPU

Two 7B models = 28GB VRAM needed
g5.xlarge has 24GB VRAM
→ Must use separate instances for blue-green deployment

This isn't just best practice - it's technically required due to VRAM constraints.

Step 2: Deployment Strategy - Blue-Green with Canary

Infrastructure Setup:

Current (1 instance)
g5.xlarge
Old Model (7B)
vLLM serving
100% Traffic →
During Deployment (2 instances)
g5.xlarge
Old Model (7B)
vLLM serving
95% Traffic
g5.xlarge
New Model (7B)
vLLM serving
5% Traffic (Canary)
⚖️ Load Balancer with Weighted Routing
After Cleanup (1 instance)
g5.xlarge
New Model (7B)
vLLM serving
100% Traffic →

Cost: 2× $1/hr = $2/hr during 4-6 hour rollout = ~$8-12 total

Step 3: Rollout Phases

Phase 1: Pre-deployment Validation (Day 1, Off Production)

1. Deploy new model to second g5.xlarge

Spin up new instance, load new model with vLLM

2. Offline Testing

  • Smoke tests: Model loads, responds correctly
  • Latency tests: p95 < 100ms target
  • Load test: Handle 100 concurrent users
  • GPU memory check: Stays under 20GB

3. Shadow Testing (CRITICAL STEP)

Route 10% of production traffic to BOTH models

→ Users see OLD model responses (no risk)

→ NEW model responses logged for comparison

→ Run for 2-4 hours, analyze differences offline

This catches issues without user-facing risk!

Phase 2: Canary Rollout (Day 2, Off-Peak Hours)

Hour 1: Old: 95% │ New: 5% ← Monitor closely Hour 2: Old: 85% │ New: 15% ← If metrics good Hour 3: Old: 50% │ New: 50% ← Confidence building Hour 4: Old: 10% │ New: 90% ← Almost there Hour 5: Old: 0% │ New: 100% ← Full cutover Keep old instance running for 24 hours (quick rollback available)

Acceptance criteria at each canary step:

  • Error rate < 0.5% (was <0.2% baseline)
  • p95 latency < 100ms (same as old model)
  • p99 latency < 150ms
  • GPU memory < 20GB (safe margin from 24GB)
  • Batch size averaging 8-32 (batching working)
  • No user complaints or support tickets

Phase 3: Cleanup (Day 3)

If no issues after 24 hours at 100%:

  • Terminate old model instance
  • Update deployment docs to new model as default
  • Archive old model for compliance
Step 4: Rollback Strategy

Automated Rollback Triggers:

IF any of these conditions:

  • Error rate > 1% (baseline was <0.2%)
  • p95 latency > 150ms (baseline was <100ms)
  • GPU OOM errors detected
  • 5xx error rate spikes > 2%

THEN automatically:

  • Alert on-call engineer
  • Shift traffic: New model → 0%
  • All traffic back to old model (~30 sec)
  • Keep new instance running for debugging

Manual Rollback (Single Command):

# At load balancer - shift all traffic back to old model aws elbv2 modify-target-group \ --target-group-arn <new-model-tg> \ --weight 0 # Takes effect in ~30 seconds # All traffic back to old model
Step 5: Monitoring & Validation

Metrics Dashboard (Old vs New Side-by-Side):

Performance Metrics
  • Latency: p50, p95, p99
  • Throughput: req/s
  • Error rate: 2xx, 4xx, 5xx
Resource Metrics
  • GPU memory usage
  • GPU utilization %
  • Batch size distribution
Quality Metrics
  • User satisfaction (if tracked)
  • Conversation completion rate
  • Response coherence scores
Business Metrics
  • User engagement time
  • Feature usage patterns
  • Support ticket volume
🎓 Key Learning Points - What Makes This ML-Specific

1. VRAM Constraint Forces Infrastructure Decision

Unlike web services where you can run multiple app versions on one server, you CANNOT fit two 7B models in 24GB VRAM (need 28GB). Blue-green requires a second instance - not optional, technically required.

2. Shadow Testing Before Canary

For web services, you might go straight to canary. For ML, run shadow testing first - both models process requests, but only old model's responses served. Validates quality without user-facing risk.

3. ML-Specific Metrics Matter

Don't just track "latency" - track p95/p99 distribution. Don't just check "it works" - check GPU memory stays <20GB, batch size maintains 8-32, and quality metrics if available.

4. Session Stickiness for Consistency

During canary, use sticky sessions so same user always gets same model. Prevents confusing UX where responses change mid-conversation. Not needed for rollback - just shift traffic to 0%.

Interview Tip: Frame deployment strategies with ML constraints (VRAM, GPU memory, inference latency) to show you understand this isn't just generic DevOps.

✅ Complete Interview Answer

"For zero-downtime upgrade with rollback capability, I'd use blue-green deployment with canary rollout."

First, the constraint: Two 7B models need 28GB VRAM, but g5.xlarge only has 24GB. So I must spin up a second instance temporarily - can't run both models on one GPU.

Phase 1 - Shadow testing: Deploy new model to second g5.xlarge. Route 10% of production traffic to BOTH models but only serve old model's responses. Compare outputs offline for 2-4 hours. This validates quality without user-facing risk.

Phase 2 - Canary rollout: Use load balancer with weighted routing. Start at 5% new model during off-peak hours, monitor for 1 hour. If metrics healthy (error rate <0.5%, p95 latency <100ms, GPU memory <20GB), progress to 15%, then 50%, 90%, 100%.

Phase 3 - Automated rollback: Set alerts on error rate spikes, latency degradation, or GPU OOM. If triggered, auto-rollback shifts traffic to 0% new model in ~30 seconds. Keep old instance running 24 hours for quick manual rollback if needed.

Cost: ~$8-12 for the 4-6 hour rollout (two instances), then back to single instance."

Scenario 3: Model Upgrade Needs Better Hardware
Intermediate

Your team wants to upgrade from the 7B model to a 13B model for better quality. The 13B model in FP16 is 26GB, which won't fit on your current g5.xlarge (24GB VRAM). Budget is a concern.

Your task: What options do you have? Consider instance types, precision formats, and trade-offs. What would you recommend?

Step 1: Calculate Requirements

13B model VRAM calculation:

Model weights: 13B × 2 bytes (FP16) = 26 GB Activations: ~3 GB Batching buffer: ~1 GB KV cache: ~3-5 GB ────────────────────────────────── Total VRAM needed: 33-35 GB
❌ Problem: Current Infrastructure Won't Work

Current: g5.xlarge has 24GB VRAM

Needed: 33-35GB VRAM

Model won't fit! Need new approach.

Step 2: Options Analysis

Option A: Upgrade to Larger Instance

g5.2xlarge

VRAM: 24GB (same as g5.xlarge!)

❌ Won't help

g5.12xlarge

VRAM: 4× 24GB = 96GB total

Cost: $5/hr = $3,600/month

✅ Fits easily

Option B: Use Lower Precision (FP8)

FP8 Optimization
13B × 1 byte (FP8) = 13 GB + Activations: ~2 GB + Batching: ~1 GB + KV cache: ~2-3 GB ────────────────────── Total: 18-20 GB ✅ Fits on g5.xlarge!

Trade-offs:

  • Memory: 50% reduction (35GB → 18GB)
  • Speed: 2x faster inference
  • Accuracy: ~5% degradation (95% of FP16 quality)
  • Cost: Same as current ($720/month)

Option C: Quantization (INT8)

Even more aggressive compression

Size: 13GB (same as FP8)

Accuracy: ~10% degradation

⚠️ Only if FP8 quality unacceptable and budget critical

Step 3: Decision Framework
Option Cost Quality Speed
FP8 on g5.xlarge $720/mo 95% of FP16 2x faster
FP16 on g5.12xlarge $3,600/mo 100% quality Baseline
INT8 on g5.xlarge $720/mo 90% of FP16 2x faster
✅ Recommended Approach: Try FP8 First

Phase 1: Deploy 13B in FP8 on g5.xlarge

  • Cost: Same as current ($720/month)
  • Speed: 2x faster than FP16
  • Quality: Test with A/B against 7B FP16

Phase 2: Validate Quality

  • Run shadow testing (both models on 10% traffic)
  • Compare user satisfaction scores
  • If 13B FP8 ≥ 7B FP16: Deploy! You win.
  • If 13B FP8 < 7B FP16: Proceed to Phase 3

Phase 3: If FP8 Quality Unacceptable

  • Upgrade to g5.12xlarge for FP16
  • Cost: 5x increase ($3,600/month)
  • Benefit: Can serve 4-5x more traffic (4 GPUs)

Rationale: Always try precision optimization before scaling hardware. FP8 gives 95% quality at 0% cost increase - worth testing!

🎓 Key Learning: Precision Optimization Before Hardware Scaling

Common mistake: Immediately upgrading to g5.12xlarge (5x cost) without trying FP8 first.

Better approach:

  1. Try FP8 on existing hardware (free to test)
  2. Validate quality with A/B testing
  3. Only upgrade hardware if quality unacceptable

Real-world result: Most production LLMs run FP8 or even INT8 with minimal quality impact. You might never need that g5.12xlarge!

Scenario 6: Latency Crisis
Intermediate

Your chatbot's p95 latency has crept up from 50ms to 200ms over the past month. Users are complaining. You can't change the model (it's the right size and quality). You need to get back to <50ms p95.

Your task: Debug and fix the latency issue. What would you investigate? What optimizations would you try?

Systematic Debugging

Latency crept up over time (50ms → 200ms) suggests degradation, not sudden change.

Most Common Causes:

1. KV Cache Memory Leak

GPU memory: 18GB → 23GB over days

Fix: Restart vLLM service

2. Reverted to PyTorch

Not using TensorRT anymore

Fix: Re-deploy .trt model

3. Batch Size Dropped

Was 32, now 1-2

Fix: Check vLLM config

4. GPU Thermal Throttling

Temperature >85°C

Fix: Check cooling

✅ Investigation Checklist
# 1. Check GPU memory over time nvidia-smi dmon -s mu # 2. Check vLLM metrics curl http://instance:8000/metrics # 3. Verify TensorRT model ls -lh /models/model.trt # 4. Check batch size config cat /etc/vllm/config.yaml

Quick Fix (Try First):

systemctl restart vllm

If latency drops to 50ms → was memory leak!

Scenario 9: Model Drift Detection
Intermediate

Your chatbot has been in production for 6 months. User satisfaction scores have dropped from 85% to 72%. The model hasn't changed, but the types of questions users ask have evolved. You're seeing more errors and "I don't know" responses.

Your task: How do you detect and address model drift? What metrics would you track? When do you retrain vs fine-tune vs replace the model?

Understanding Model Drift

Model hasn't changed, but data distribution has:

6 months ago (training data):

"What's the weather?" "Set a reminder" "Tell me a joke"

Now (production queries):

"How do I use the new AI features?" "What's the difference between Pro and Enterprise?"

Model never trained on product-specific questions → high "I don't know" rate!

✅ Three-Phase Solution

Immediate (Week 1): Add RAG

Retrieval Augmented Generation

Add product docs, FAQs to knowledge base

Model gets current context without retraining

Short-term (Month 1): Fine-tune with LoRA

Collect recent 30 days of queries

Fine-tune on new question patterns

Cost: $1,000-5,000, Time: 1 week

Long-term (Month 2-3): Full Retrain

80% original training data + 20% recent production

Cost: $50,000+, Time: 2-4 weeks

Prevention: Quarterly retraining schedule to catch drift early!

Scenario 10: Security & Compliance
Advanced

Your company is deploying the chatbot for healthcare use (HIPAA compliance required). The model needs to run in a private VPC with no internet access. You also need to implement RAG (Retrieval Augmented Generation) to pull from a private medical knowledge base.

Your task: Design a secure, compliant deployment. How do you handle model downloads, RAG integration, and logging while meeting regulatory requirements?

HIPAA Architecture (Private VPC)

Private VPC - No Internet Gateway

Components:
  • g5.xlarge with encrypted EBS volumes
  • 7B Model (vLLM) in private subnet
  • Vector DB (Weaviate) for medical knowledge base
  • Internal load balancer (no public IP)
  • VPC endpoints for S3, CloudWatch (no internet)

Access: VPN or Direct Connect ONLY

No public internet access to or from VPC

✅ HIPAA Compliance Checklist

Encryption:

  • At rest: KMS encryption for EBS, S3, RDS
  • In transit: TLS 1.3 for all connections
  • PHI: All patient data encrypted

Access Controls:

  • IAM roles with least privilege
  • MFA required for all users
  • Audit logs (7-year retention for HIPAA)

Model Deployment (No Internet):

  • Upload model to S3 from internet-connected env
  • Download via S3 VPC endpoint (stays in AWS network)
  • Or use pre-baked AMI with model included

RAG Implementation:

  • Self-hosted Weaviate in same VPC
  • Medical knowledge base encrypted
  • All queries logged for audit

Additional Costs: ~$1,000-2,000/month for compliance overhead
